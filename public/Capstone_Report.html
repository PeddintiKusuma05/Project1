ABSTRACT

Abstract

Accessing appropriate medical care is often hindered by uncertainty about which specialist to consult and by fragmented digital experiences. This project presents a client-side web application that transforms patient-reported symptoms into actionable guidance and tailored doctor recommendations. Built with React, TypeScript, Vite, Tailwind CSS, and reusable UI primitives, the system streamlines the pathway from symptom entry to contacting a relevant healthcare provider. The prototype demonstrates an end-to-end user flow encompassing symptom input, analysis visualization, ranked recommendations, role-aware dashboards (patient and doctor), secure navigation via an authentication context, and supportive content such as FAQs. While the current implementation runs fully on the client and uses heuristic logic for demonstrative recommendations, the architecture is designed for integration with secure backend services and standards-compliant data management. This report details the motivation, literature background, methodology, results, and future directions, emphasizing usability, privacy, and maintainability in digital health.

Empirical evaluation centers on UX principles, information architecture, and responsive design. The results indicate that a clear, mobile-first interface coupled with transparent guidance reduces friction and improves user confidence in navigating care options. Limitations include the absence of clinical validation and production-grade security controls—both prioritized in the proposed future scope, which includes ML/NLP-based triage models, E2E encryption, federated identity, audit logging, and integration with provider directories and scheduling systems.

Keywords: symptom triage, doctor recommendation, web application, React, TypeScript, digital health, usability, privacy

1. Introduction

1.1 Background and Motivation

Patients frequently struggle to determine which specialist to consult when experiencing new or ambiguous symptoms. Online resources can be overwhelming or unreliable, and healthcare systems often lack clear pathways that translate layperson descriptions into clinical categories. This project addresses the gap by offering an approachable interface that captures symptom narratives and produces clear, non-diagnostic guidance alongside appropriate specialist recommendations.

1.2 Objectives

  • Capture user symptoms efficiently with guided inputs and validation.
  • Provide interpretable analysis to bridge lay and clinical terminology.
  • Recommend relevant specialists with actionable next steps to contact.
  • Support role-aware dashboards for patients and doctors.
  • Ensure secure, maintainable, and accessible implementation.

1.3 Scope

The present work implements a fully client-side SPA prototype using React and TypeScript, styled with Tailwind CSS and composed with accessible UI primitives. Authentication is modeled via a context provider and protected routes, while recommendation logic is demonstrated heuristically. The architecture is designed for incremental integration of production services and data sources.

1.4 Document Structure

The report proceeds with a review of literature, a detailed methodology of system design and implementation, empirical results and discussion, and concludes with future scope.

2. Review of Relevant Literature

2.1 Symptom Checkers and Triage Systems

Prior research on digital symptom checkers underscores the balance between usability and clinical safety. Explainability and clarity of limitations are critical to preventing overreliance. Systems that position outputs as guidance rather than diagnosis align with ethical practice and user expectations.

2.2 Patient–Physician Matching

Studies on care navigation show that filtering by specialty, availability, and patient preference improves uptake of consultations. Transparent criteria in ranking bolster trust. Integrations with scheduling reduce friction and improve conversion from interest to appointment.

2.3 UX Patterns in Health Applications

Progressive disclosure, concise copy, and mobile-first layouts are repeatedly found to reduce drop-off. Accessibility following WCAG guidelines is essential to ensure inclusive access, especially where health literacy varies widely.

2.4 Privacy, Security, and Compliance

Production healthcare software must conform to legal frameworks such as HIPAA and GDPR. Key practices include encryption, least-privilege access, audit logging, consent management, and data minimization. While the current prototype is client-side, the target roadmap incorporates these principles.

2.5 Summary

Literature informs the project’s emphasis on guidance over diagnosis, transparency in recommendations, and privacy-centric design. These elements shape the system’s architecture and UX decisions.

3. Methodology

3.1 System Architecture

The application is structured as a SPA using React with TypeScript. Pages define primary routes such as Symptom Input, Analysis Result, and Doctor Recommendations. Reusable UI primitives are sourced from a component library to ensure consistency and accessibility. Styling leverages Tailwind’s utility classes.

3.2 Components and State Management

Global authentication is handled through a context provider with protected routes restricting access to sensitive pages. A lightweight store module centralizes ephemeral state such as current analysis or user role. Hooks encapsulate responsive and feedback behaviors (e.g., toast notifications, mobile detection).

3.3 Data Flow

Users input symptom descriptions which are transformed into intermediate categories via client-side logic. These categories seed recommendation generation, ranking specialists according to configurable rules. The system presents results with contextual information and prompts users to contact suitable providers.

3.4 UI/UX Design

The interface prioritizes clarity and brevity, employing progressive disclosure to guide users. Key interactions include explainers and CTAs that emphasize next steps while clarifying that guidance is not a diagnosis.

3.5 Security and Privacy Considerations

While the prototype is client-only, the design anticipates secure extensions: token-based authentication, encrypted transport, and privacy notices. Access control follows least privilege and separates patient and doctor capabilities.

3.6 Implementation Details

  • Build tooling via Vite for rapid development and optimized bundles.
  • TypeScript for type safety and clearer contracts between components.
  • Linting and formatting for consistent code quality.
  • Routing to map user journeys and enforce guards for protected pages.

4. Results and Discussions

4.1 Prototype Outcomes

The prototype demonstrates a coherent flow from symptom entry through analysis to recommendations, with users able to contact a doctor via a clear CTA. Dashboards differentiate capabilities based on user role, while the interface adapts responsively to mobile and desktop.

4.2 Usability Insights

Heuristic evaluation suggests that streamlined forms and immediate feedback improve completion rates. Clear disclaimers preserve user trust and establish appropriate expectations.

4.3 Limitations

The absence of a backend limits data persistence, auditability, and clinical validity. The recommendation logic is illustrative and requires expert curation or data-driven modeling for production contexts.

4.4 Discussion

The results validate that a modular, client-first approach can quickly realize user value and inform subsequent investment in services like identity, analytics, and provider directories. It also emphasizes the importance of explainability and privacy from the outset.

5.1 Conclusions

A client-side SPA can effectively guide patients from symptom description to actionable recommendations. The architecture supports maintainability and creates a solid base for integrating production-grade services.

5.2 Future Scope

  • Integrate secure authentication (OAuth/OIDC) and server-side validation.
  • Implement ML/NLP-based triage with model versioning and explainability.
  • Connect to provider directories, scheduling APIs, and messaging.
  • Adopt compliance controls (HIPAA/GDPR), audit logs, and consent flows.
  • Establish testing, CI/CD, observability, and feature flagging.

The prototype validates core user value and informs a phased roadmap toward a secure, scalable, and clinically useful digital health platform.

Appendices

A. Architecture Details

The React component tree is organized by pages and shared components. Each page focuses on a single responsibility (e.g., symptom input, results display), while shared UI primitives provide consistent behavior across the application. The store module centralizes ephemeral state and avoids prop drilling, and custom hooks encapsulate common behaviors.

Routing is configured to provide clear navigation paths with guards for protected resources. Error boundaries and not-found routes improve resilience and user feedback in exceptional states. Styling with Tailwind enables rapid iteration while adhering to a consistent visual language.

Build configuration (Vite) accelerates local development with HMR and efficient bundling. TypeScript configuration enforces strictness to reduce runtime errors.

The UI library (shadcn/ui) supplies accessible primitives such as dialogs, accordions, and forms. These facilitate coherent information architecture and predictable interactions.

B. Expanded Methodology

The methodology iteratively prototypes user journeys, validates usability assumptions, and refines interactions based on feedback. Accessibility checks ensure keyboard navigation, sufficient contrast, and descriptive labels. Performance budgets aim to keep the bundle light for mobile data contexts.

Security considerations include threat modeling for common web risks, separation of concerns, and preparedness for future backend integration with token-based flows and encrypted storage.

Observability plans encompass logging, error tracking, and basic metrics to inform future reliability improvements.

C. Extended Results Notes

While formal clinical validation is beyond scope, the prototype clarifies requirements for data models, recommendation explainability, and safe-guard text. It also highlights the importance of clear handoffs to human care.

Future trials should measure user comprehension, time-to-action, and conversion to booked consultations, comparing different explanation styles and CTAs.
